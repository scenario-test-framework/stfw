# 変更要望: Warn を一級実行ステータスとして導入する

## 背景・動機

シナリオテストの運用には 2 つのモードがある。

1. 回帰テスト: 完全一致を確認したい。比較不一致で Error 停止してほしい(現状の挙動そのもの)。
2. 機能変更の差分確認: 狙い通りの差分が全シナリオで発生することを検出したい。
   比較不一致を Warn として最後まで実行を進めた後、比較 NG を鳥瞰したい(現状は実現不可)。

現状は compare プラグインが比較不一致(compare-files exit 3)を一律 Error(exit 6)に変換し、
実行は全階層で停止・後続は Blocked となる(SPEC-014-02「比較不一致はステップ失敗」)。
プロセスプラグイン RC 契約の 3=Warn は体系上の予約であり、ランナーの成否判定は「0 か否か」で
3 も Error と同じ扱いになっている。

## 要望

1. プロセスプラグインのリターンコード 3(Warn)に実行意味論を与える。
   該当ステップ/プロセスを Warn として記録し、後続の実行は止めない。
   Error(6 等の非 0・非 3)は従来どおり停止 + 後続 Blocked 伝播。
2. 上位階層(process / bizdate / scenario / run)の実行ステータス集約は
   Error > Warn > Success の優先度で行う。子に Warn があれば親も Warn になる。
3. stfw run の終了コードは、全 Success=0 / Warn あり(Error なし)=3 / Error あり=6 とする。
   CI で「差分あり」を終了コードで検知できるようにする。
4. stfw status・HTML レポート・OTLP トレースで Warn を表示する(黄系の色)。
   レポートがそのまま「比較 NG の鳥瞰」ビューになる。
   OTel のスパンステータスには Warn 相当がない(Ok/Error/Unset のみ)ため、
   Warn は Ok + 属性(stfw の status 属性)で表現する。
5. compare プラグインに設定キー on_mismatch: error(既定)| warn を追加する。
   既定 error で回帰テスト運用との後方互換を維持する。
   warn のとき compare-files の比較不一致(exit 3)を exit 3 のまま返す(6 に変換しない)。
6. SPEC-014-02「比較不一致はステップ失敗(一律 Error 停止)」を改訂し、
   on_mismatch 設定による選択制(error=従来挙動 / warn=記録して続行)とする。

## 制約・互換性

- プロセスプラグイン実行契約(RC 0/3/6)自体は変えない。意味論の追加のため既存プラグインは
  壊れない(ただし 3 を返していたプラグインは「止まらなくなる」挙動変化がある。
  リリースノートで明示する)。
- ジャーナルの status に "Warn" が増える。旧バージョンの run のジャーナル(Warn なし)と
  混在しても、report 再生成・housekeep・status が壊れないこと。
- teardown フックへ公開する stfw_run_status に Warn が入るケースを定義する
  (Warn あり・Error なしの run では stfw_run_status=Warn)。
