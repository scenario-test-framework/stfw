stfw に組み込みプロセスプラグイン群を追加し、回帰テストの「データ準備・実行・エビデンス収集・
期待値比較」を全自動化できるようにしたい。

## 背景・ポジショニング（REQ-008 の理由の具体化でもある）

stfw は回帰テストシナリオの全自動実行ツールである。grafana k6 等の「準備された環境と
リクエスト/レスポンス」を前提にした仕組みと異なり、実行前後の作業（データ準備・後片付け・
エビデンス収集・期待値比較）や、画面のないバッチ処理の起動までをサポート範囲とする。

1 つの業務日付（bizdate）は Arrange（準備）→ Act（実行）→ Collect（収集）→ Assert（検証）の
パイプラインであり、組み込みプラグインはこの各フェーズの汎用部品を提供する。
プロダクト固有の知識（業務日付の更新方法・ジョブスケジューラの呼び出し方・共通マスターデータ）は
利用者がカスタムプラグインとして組み込みプラグインを組み合わせて実装する 2 層構造とする。

## 追加する組み込みプラグイン

### 収集系（Collect）

1. collectLog — テスト対象ホストからの実行ログ収集
   - scenario-test-framework/logfilter（別リポジトリの OSS バイナリ）を利用する
   - 設定: 収集先ホストグループ（inventory 参照）ごとの収集ファイルパス正規表現リスト
   - 処理: scp で logfilter バイナリを収集先ホストへ転送 → ssh で実行し
     「bizdate の実行開始日時」以降のログ行にフィルタ → scp でローカルへ収集 → 転送したバイナリを削除
   - フィルタ基準時刻は実行ジャーナルの bizdate node_start イベント時刻を使用し、
     プラグイン env 契約（stfw_bizdate_start_ts 等）として公開する
   - 前提: 収集先ホストとの時刻同期（ドキュメントに明記する制約）
2. collectFile — テスト対象ホストからのファイル収集（外部 IF ファイル等）
   - 設定: 収集先ホストグループ（inventory 参照）ごとの収集ファイルパス正規表現リスト
   - 処理: scp で収集先ホストのファイルをローカルのエビデンスディレクトリへ収集

### データストア系（Arrange / Collect）

3. RDBMS 系: exportMysql / importMysql / clearMysql、exportPostgres / importPostgres / clearPostgres
   - export: テーブル名リストを指定してヘッダー付き CSV でエクスポート（エビデンス収集と期待値作成に使用）
   - import: export が出力した形式（ヘッダー付き CSV）をインポート（export/import はラウンドトリップ可能）
   - clear: テーブル名リストを指定して truncate
4. KVS 系: RDBMS 系と同じバリエーション（export / import / clear）
5. 対象製品は代表的な OSS のみとし、public cloud のマネージド製品はまだ対象にしない
   - RDBMS: MySQL（MariaDB は MySQL 互換として同一プラグインでサポート）、PostgreSQL
   - KVS: Redis（Valkey は Redis 互換として同一プラグインでサポート）
   - 将来の製品追加はプラグイン追加で対応する

### 検証系（Assert）

6. compare — エビデンスと期待値の比較
   - scenario-test-framework/compare-files（別リポジトリの OSS）を利用する
   - 設定: compare-files の起動設定・比較レイアウトは stfw プロジェクトごとに持たせる
   - expect ディレクトリは収集系プラグイン（collectFile / exportXxx）の出力ディレクトリ構造と
     同型にする（エビデンスディレクトリ規約）
   - 処理: actual ディレクトリを組み立て（収集系 process ディレクトリへの symlink または一時コピー）
     → compare-files を実行 → 比較結果を出力。actual / result ディレクトリは gitignore 対象
   - 比較不一致はステップ失敗（エラー時停止・Blocked 伝播の対象）とする

### 実行系（Act）

7. invokeWeb — grafana k6（ブラウザモード）による画面での取引入力と画面操作での検証
8. invokeRest — grafana k6 による API での取引入力とレスポンス検証

## 横断的な設計決定

- 接続情報の持ち方: 収集先・接続先ホストは既存の inventory（ホストグループ）を参照し、
  資格情報は secret（age 暗号化）、SSH ホストキーは ssh trust を利用する。
  プラグイン設定には接続情報を直接書かず、グループ名参照のみとする
- エビデンスディレクトリ規約の新設: 収集系プラグインの出力構造 = compare の expect 構造となる
  ディレクトリ命名規約を定め、ディレクトリ規約・プラグイン env 契約に次ぐ第 3 の互換境界として扱う
  （具体的な命名は設計フェーズで確定）
- ランタイム依存の宣言: プラグインごとに前提コマンド（k6・mysql/psql クライアント・ssh/scp 等）を
  宣言し、stfw validate が存在チェックで検出する。Docker イメージは既存の最小構成に加えて
  依存全部入りのタグ（例: stfw:full）を提供する

## 利用者が用意するカスタムプラグインの想定（ドキュメント化対象のガイドパターン）

- updateBizDate: bizdate ディレクトリの日付に合わせてテスト対象 DB の業務日付を更新する
  （import 系 CSV を動的生成して importXxx を呼ぶ）
- invokeJob: テスト対象プロダクトのジョブスケジューラを呼び出し方に合わせてキックする
- importMaster / export / clear: シナリオ共通のマスターデータを importXxx 系の呼び出しとして定義する

## 想定シナリオ（例: 受注から入金）

業務日付ごとに Arrange → Act → Collect → Assert を反復し、日次（updateBizDate → clear →
importMaster → import → invokeWeb → invokeRest → invokeJob(日次締め) → collectFile →
collectLog → export → compare）に加え、月末に月次処理、四半期末・半期末・年度末に
それぞれの締め処理を呼ぶ。会計年度をまたぐ長時間実行となるため、HTML レポート・
OTel トレースによる準リアルタイム監視と組み合わせて利用する。

## 影響範囲（想定）

- 要求: REQ-008（プラグイン拡張）の理由をこの構想で具体化。組み込みプラグイン群
  （収集系・データストア系・検証系・実行系）とエビデンスディレクトリ規約・依存宣言は
  新規要求として追加
- RDRA: 情報「エビデンス」「期待値（expect）」等の追加、外部システム「テスト対象ホスト群」への
  scp/ssh 連携イベントの具体化、k6・logfilter・compare-files 等の外部ツール、
  条件「エビデンスディレクトリ規約」「時刻同期前提」「比較不一致はエラー」等の追加、
  BUC「シナリオ作成」「シナリオ一括自動実行」「結果確認」のアクティビティ拡充
- 本要望は要求定義のみ（実装は含まない）
